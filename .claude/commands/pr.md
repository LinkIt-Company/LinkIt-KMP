현재 브랜치의 변경사항을 분석하고, PR 템플릿에 맞춰 Pull Request를 작성하여 GitHub에 올린다.

## 진행 순서

1. **변경사항 파악**: 아래 명령어들을 병렬로 실행하여 현재 상태를 파악한다.
   - `git status` — 변경/추가된 파일 확인
   - `git log main..HEAD --oneline` — 현재 브랜치의 커밋 목록 확인
   - `git diff main...HEAD --stat` — main 대비 변경된 파일 통계 확인
   - `git diff main...HEAD` — 실제 변경 내용 확인
   - `git branch --show-current` — 현재 브랜치명 확인

2. **PR 템플릿 읽기**: `.github/PULL_REQUEST_TEMPLATE.md` 파일을 읽어서 PR 본문 형식을 파악한다.

3. **관련 이슈 파악**: 브랜치명이나 커밋 메시지에서 이슈 번호를 추출한다. 이슈 번호를 찾을 수 없으면 사용자에게 "관련 이슈 번호가 있나요? 없으면 엔터를 눌러주세요." 라고 질문한다.

4. **PR 초안 작성**: 변경사항을 분석하여 PR 템플릿의 각 섹션을 채운다.
   - **Issue Number**: 파악된 이슈 번호 기입 (closes #N)
   - **Changes**: 커밋들과 diff를 분석해 주요 변경사항을 한국어 bullet point로 정리. 의미 있는 변경이 있을 경우 As-Is / To-Be 비교 형식으로 작성하여 무엇이 어떻게 바뀌었는지 명확히 전달한다.
   - **Review Point**: 리뷰어가 중점적으로 봐야 할 부분을 구체적으로 작성
   - **Architecture / Flow** (Optional): 변경사항의 이해를 돕기 위해 Mermaid 다이어그램으로 시각화한다. 아래 판단 기준을 참고하여 필요한 경우에만 포함한다.
   - **Test Plan**: 어떻게 테스트했는지, 혹은 테스트 방법을 작성
   - **Screenshot**: UI 변경이 있는 경우에만 포함 (없으면 섹션 제거)
   - **관련 Reference 자료**: 관련 자료가 있는 경우에만 포함 (없으면 섹션 제거)

5. **초안 확인**: 작성한 PR 초안을 사용자에게 보여주고, "이대로 PR을 생성할까요? 수정할 부분이 있으면 말씀해주세요." 라고 확인을 요청한다.

6. **PR 생성**: 사용자가 승인하면 아래 단계를 진행한다. 수정 요청이 오면 반영 후 다시 확인을 요청한다.
   - 커밋되지 않은 변경사항이 있으면 사용자에게 알리고 커밋 여부를 확인한다.
   - `git push -u origin <branch>` 로 리모트에 푸시한다.
   - `gh pr create --title "제목" --body "본문" --base main` 으로 PR을 생성한다.
   - 생성된 PR URL을 사용자에게 알려준다.

7. **코드 리뷰 코멘트 작성**: PR 생성 후, diff를 다시 분석하여 핵심 변경 포인트나 보충 설명이 필요한 코드에 Review Comment를 남긴다.
   - `gh api repos/{owner}/{repo}/pulls/{pr_number}/comments` 를 사용하여 특정 파일의 특정 라인에 코멘트를 남긴다.
   - 코멘트 대상 선정 기준:
     - 설계 의도나 선택 이유를 설명해야 하는 코드 (예: 왜 이 패턴을 썼는지)
     - 리뷰어가 놓칠 수 있는 핵심 로직 변경
     - 기존 코드와 달라진 방식이나 컨벤션
     - 사이드 이펙트가 있을 수 있는 부분
   - 코멘트하지 않는 경우:
     - 자명한 변경 (import 추가, 포맷팅, 설정값 변경 등)
     - PR 본문의 Changes에서 이미 충분히 설명한 내용
   - 코멘트 형식: 한국어로 간결하게 작성. 필요 시 코드 블록(`````)을 활용하여 예시를 보여준다.

## 규칙
- PR 제목은 70자 이내로 간결하게 작성한다. 한국어로 작성한다.
- PR 본문은 한국어로 작성한다.
- Optional 섹션은 해당 내용이 없으면 과감히 제거한다.
- PR 본문은 HEREDOC을 사용하여 전달한다.
- base 브랜치는 기본적으로 `main`을 사용한다.
- 사용자가 최종 승인하기 전까지 절대 push나 PR 생성을 하지 않는다.

## Mermaid 다이어그램 가이드

### 포함하는 경우
- 모듈/레이어 간 의존 관계가 변경되었을 때 → `graph` 또는 `classDiagram`
- 새로운 화면 흐름이나 네비게이션이 추가되었을 때 → `flowchart`
- API 호출, 이벤트 처리 등 비동기 흐름이 복잡할 때 → `sequenceDiagram`
- 상태 전이가 핵심인 변경일 때 → `stateDiagram-v2`

### 포함하지 않는 경우
- 설정 파일, 문서, 템플릿만 변경한 경우
- 단순 버그 수정, 리네이밍, 포맷팅
- 파일 1~2개의 소규모 변경
- 다이어그램 없이도 Changes 섹션만으로 충분히 이해 가능한 경우

### 작성 원칙
- 한 PR에 다이어그램은 최대 2개까지만 사용한다.
- 노드/단계는 10개 이내로 핵심만 담는다.
- 이번 PR에서 변경된 부분을 시각적으로 구분한다 (예: 스타일링, 주석 등).
- 구조적 변경이 있을 경우, As-Is / To-Be 다이어그램을 나란히 배치하여 변경 전후를 비교한다.
