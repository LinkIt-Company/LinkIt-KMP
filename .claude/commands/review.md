$ARGUMENTS 번호의 PR 코드 변경사항을 분석하고 코드 리뷰를 수행한다.

## 진행 순서

1. **PR 정보 수집 및 브랜치 체크아웃**: 아래 명령어들을 병렬로 실행하여 PR 정보를 파악한다.
   - `gh pr view $ARGUMENTS --json title,body,state,files,headRefName` — PR 메타데이터 및 변경 파일 목록 확인
   - `gh pr diff $ARGUMENTS` — 코드 변경사항 확인
   - PR 브랜치를 로컬에 가져와 체크아웃한다: `git fetch origin <headRefName> && git checkout <headRefName>`
     - 이렇게 해야 PR에서 새로 추가된 파일을 직접 읽을 수 있다.

2. **프로젝트 구조 파악**: 변경된 파일이 속한 모듈과 주변 코드를 읽어서 기존 패턴과 컨벤션을 파악한다.

3. **코드 리뷰 수행**: 아래 리뷰 기준에 따라 변경사항을 분석한다.

4. **리뷰 결과 출력**: 발견된 사항을 카테고리별로 정리하여 사용자에게 보여준다.
   - 직접 수정 가능한 항목은 🔧 태그를 붙이고, 구체적인 수정 코드를 코드 블록으로 함께 제시한다.

5. **코드 수정 제안**: 🔧 항목이 있으면 "수정 가능한 항목이 N개 있습니다. 코드를 직접 수정할까요?" 라고 사용자에게 확인한다.
   - 사용자가 승인하면 해당 PR의 브랜치를 체크아웃하고 코드를 수정한다.
   - 수정 후 변경사항을 사용자에게 보여주고 커밋 여부를 확인한다.

6. **PR 리뷰 제출**: 리뷰 결과 출력 후 자동으로 `gh api` 를 사용하여 발견 사항 하나당 하나의 인라인 코멘트를 해당 파일의 해당 라인에 남긴다.
   - `gh api repos/{owner}/{repo}/pulls/$ARGUMENTS/comments` 를 사용한다.
   - 각 코멘트의 필수 필드: `body`, `commit_id` (PR의 HEAD commit SHA), `path` (파일 경로), `line` (diff 내 라인 번호), `side` ("RIGHT")
   - `commit_id`는 `gh pr view $ARGUMENTS --json headRefOid --jq '.headRefOid'` 로 조회한다.
   - PR 작성자는 `gh pr view $ARGUMENTS --json author --jq '.author.login'` 으로 조회한다.
   - 각 코멘트 본문은 첫 줄에 `@{작성자}` 멘션만 작성하고, 줄바꿈(`\n\n`) 후 심각도 태그와 설명을 작성한 뒤, 끝에 `\n\n> 🤖 Claude Code가 작성한 리뷰입니다.` 를 추가한다.
   - 👏 Good 항목은 인라인 코멘트 대신 `gh pr review $ARGUMENTS --comment --body "내용"` 으로 전체 코멘트로 남긴다.

## 리뷰 기준

### 1. 정확성 (Correctness)
- 버그 또는 잠재적 버그
- 엣지 케이스 미처리
- 널 안전성 (nullable 처리 누락)
- 오프바이원 에러
- 에러 핸들링 누락

### 2. 설계 (Design)
- 단일 책임 원칙 위반
- 불필요한 결합도
- 의존성 방향 (상위 레이어가 하위 레이어에 의존하는지)
- 인터페이스/추상화 적절성

### 3. 프로젝트 구조 일관성 (Consistency)
- 모듈 구조 및 패키지 컨벤션 준수 여부
- 아키텍처 레이어 규칙 (data/domain/presentation 등) 준수 여부
- 기존 코드와 동일한 패턴 사용 여부 (DI, Repository, UseCase 등)
- 네이밍 컨벤션 (파일명, 클래스명, 함수명) 일관성
- 새 파일/클래스의 위치가 기존 구조와 일치하는지

### 4. 성능 (Performance)
- 불필요한 리컴포지션 유발 (Compose)
- N+1 쿼리 또는 불필요한 반복 호출
- 메모리 누수 가능성 (CoroutineScope, Flow 구독 등)

### 5. 가독성 (Readability)
- 불명확한 네이밍
- 과도한 복잡도 (깊은 중첩, 긴 함수)
- 매직 넘버/스트링
- 코드 중복

### 6. 보안 (Security)
- 하드코딩된 시크릿/키
- 입력 검증 누락
- 인젝션 가능성

### 7. KMP 특화 (Kotlin Multiplatform)
- expect/actual 선언 일관성
- 플랫폼별 사이드이펙트 고려
- 공통 모듈에 플랫폼 의존 코드 포함 여부

## 리뷰 결과 출력 형식

각 발견 사항을 아래 형식으로 정리한다:

```
## 리뷰 결과

### 요약
- 전체적인 변경사항에 대한 한줄 평가
- 주요 발견 사항 개수: N개

### 발견 사항

#### [심각도] 카테고리 — 파일명:라인
설명과 개선 제안

#### 🔧 [심각도] 카테고리 — 파일명:라인
설명과 수정 코드 (직접 수정 가능한 항목)
```

심각도는 다음 4단계로 구분한다:
- 🔴 **Must Fix**: 버그, 보안 이슈, 데이터 손실 가능성
- 🟡 **Should Fix**: 설계 문제, 성능 이슈, 구조 불일치
- 🟢 **Suggestion**: 가독성, 컨벤션, 개선 제안
- 👏 **Good**: 잘 작성된 코드, 좋은 설계 판단, 모범 사례 적용

## 규칙
- 자명한 변경(import 정리, 포맷팅, 설정값)에는 코멘트하지 않는다.
- 문제를 지적할 때는 반드시 개선 방안을 함께 제시한다.
- 칭찬할 만한 좋은 코드가 있으면 언급한다.
- 리뷰는 한국어로 작성한다.
- PR의 맥락(제목, 본문, 관련 이슈)을 고려하여 리뷰한다.
